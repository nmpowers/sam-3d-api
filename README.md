# SAM 2 + SAM 3D Ticketing API

> ⚠️ **Disclaimer:** This project is a work in progress and is not production-ready. It is intended as an exploratory project to test, utilize, and build upon Meta's SAM (Segment Anything) and SAM 3D Objects models. Expect bugs, edge cases, and future architectural changes.

This repository provides a FastAPI-based backend that unifies Meta's **SAM 2** (for high-quality 2D image segmentation) and **SAM 3D Objects** (for 2D-to-3D generation). It uses an asynchronous ticketing system to handle heavy GPU workloads, allowing users to upload images, generate masks, and export 3D models in both `.glb` (Mesh) and `.ply` (Gaussian Splat) formats.

The exported `.glb` meshes are highly optimized for AR/VR workflows and native game engines, while the `.ply` splats provide high-fidelity web previews.

## Architecture & Environment Separation

Meta's SAM 2 and SAM 3D Objects require vastly different and often conflicting dependencies (e.g., different PyTorch versions, strictly pinned CUDA extensions like Kaolin and `nvdiffrast`). 

To solve this, the Docker image builds **two completely isolated Python environments**:

1. **Environment A (The API - Base Env):** - Runs Python 3.12 and modern PyTorch.
   - Hosts the FastAPI server (`api.py`).
   - Keeps the SAM 2 model loaded in VRAM for instant 2D point/box segmentation.
2. **Environment B (The Worker - Conda Env):** - A strict Conda environment (`sam3d-objects`) matching Meta's exact 3D training environment.
   - Runs `worker_3d.py` to execute the heavy 3D generation pipeline.

When a 3D generation task is requested, the API spawns a subprocess that explicitly calls the Conda environment's Python executable, ensuring the 3D pipeline runs without dependency collisions.

### Core Scripts
* `api.py`: The FastAPI server handling incoming requests, serving SAM 2 inference, and managing the task queue.
* `worker_3d.py`: The isolated worker script that processes the segmented image through the SAM 3D pipeline and exports the `.glb` and `.ply` files.
* `download_models.py`: A pre-flight script that downloads the necessary Hugging Face and Meta checkpoints into the container.
* `launcher.py`: The entry point script that starts the API.

## API Endpoints

* **`POST /segment`**
  * Accepts a Base64 image and `(x, y)` coordinates (or bounding box).
  * Returns a Base64 mask generated by SAM 2.
* **`POST /generate-3d`**
  * Accepts a Base64 image and Base64 mask.
  * Spawns the worker process and returns a `task_id`.
* **`GET /status/{task_id}?format=glb|ply`**
  * Polls the status of the background task.
  * If completed, serves the requested 3D file format as an `application/octet-stream` download.

## Local Linux Setup (Building the Docker Image)

To run this locally, you need a Linux machine with an NVIDIA GPU, Docker, and the NVIDIA Container Toolkit installed. At least 24GB of VRAM (e.g., RTX 3090 / 4090) is highly recommended for the 3D generation step.

1. **Clone the repository:**
   ```bash
   git clone <your-repo-url>
   cd <your-repo-directory>
   
2. Build the Docker Image:
  You will need to pass your Hugging Face token as a build argument so the script can download the SAM 3D checkpoints.

  ```bash
  docker build --build-arg HF_TOKEN="your_hf_token_here" -t sam-3d-api -f Dockerfile.seperated .
  ```

3. Run the Container:
  Ensure you pass the --gpus all flag so the container can access your hardware.

  ```bash
  docker run --gpus all -p 8000:8000 -it sam-3d-api
  ```

  The API will now be accessible at http://localhost:8000.

## Cloud / Local Deployment Guide
If you do not have local access to a high-end GPU, you can deploy this Docker image on a cloud service such as RunPod, etc.

1. Prepare the Image
   - You must first push your built Docker image to a container registry like Docker Hub or GitHub Container Registry (GHCR) so RunPod can pull it.

  ```bash
  docker tag sam-3d-api yourusername/sam-3d-api:latest
  docker push yourusername/sam-3d-api:latest
  ```

2. Create the Pod or Environment
   - You will need a GPU with at least 24GB VRAM (e.g., RTX 3090, RTX 4090, or A5000).
   - You will need at least 20GB of free disk storage as well.
   
4. Exposing the API to your Frontend
   - Once the pod or local environment is running, you can connect to the API using the resulting CloudFlare Test Tunnel.
   - These test tunnels are being used for quick testing at the moment, but you can also setup a Cloudflare config to use a different tunnel.
